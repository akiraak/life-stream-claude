# ログインなしで使えるようにする — 変更プラン

## 背景・目的

現状、モバイル / Web クライアントはどちらも起動直後にログイン画面へ遷移し、Magic Link (メール OTP) で認証しないと一切使えない。
サーバ側も `/api/shopping` `/api/dishes` `/api/saved-recipes` `/api/claude` `/api/recipes` すべてが `requireAuth` ミドルウェアの後ろにある（`server/src/index.ts:67-73`）。

App Store / Google Play の審査やユーザー初回体験の観点で「まず触ってみて価値を感じてから登録する」導線が欲しい。
本プランでは **どこまで未ログインで使えるようにするか** の案を複数提示し、採用案に応じた実装方針・ログイン導線を整理する。

---

## 現状の機能整理

| 機能 | 画面 | API | AI コスト | 他ユーザーとの共有 |
|------|------|-----|-----------|------------------|
| 買い物リスト（食材の追加・編集・チェック・並べ替え） | `(tabs)/index.tsx` | `/api/shopping`, `/api/dishes` | なし | なし（自分専用） |
| 料理単位のグルーピング | `(tabs)/index.tsx` | `/api/dishes` | なし | なし |
| AI 具材・レシピ提案（Gemini） | `components/dishes/IngredientsScreen` | `/api/dishes/:id/suggest` ほか | **あり** | なし |
| 自分のレシピ（いいね保存） | `(tabs)/recipes.tsx` | `/api/saved-recipes` | なし | なし |
| みんなのレシピ（全ユーザー公開） | `(tabs)/shared.tsx` | `/api/saved-recipes/shared` | なし | **あり（公開）** |
| いいね | 各レシピ画面 | `/api/saved-recipes/:id/like` | なし | 集計は公開 |
| 管理画面 | Web `/admin` | `/api/admin` | なし | — |

ポイント:
- **AI 呼び出しは有料** — ただし「ログインしたら無制限」ではなく、**ログイン済みでも 1 日あたり回数制限**を設ける方針（コスト青天井を防ぐため）。未ログイン時はさらに厳しい制限とする
- **買い物リスト・料理・自分のレシピは完全に個人データ** — 端末内に持てば済む
- **みんなのレシピは公開データ** — 閲覧だけなら誰でも見せてよい

## AI 呼び出しのレート制限設計（全案共通）

### 基本ルール

| ユーザー種別 | 識別キー | 1 日あたり上限（初期値・仮） | 上限超過時 |
|------------|---------|----------------------------|-----------|
| 未ログイン | 端末 ID（IDFV / SSAID のハッシュ） | **3 回/日** | ログイン誘導モーダル |
| ログイン済み | user_id | **20 回/日** | 明日また使えます表示 |
| 管理ユーザー | user_id | 無制限（or 大きい値） | — |

初期値は仮置き。運用ログを見て調整する。

### 端末 ID の取得（Expo SDK 54 調査結果）

- `expo-application` パッケージを追加（SDK 54 で公式サポート）
  - iOS: `Application.getIosIdForVendorAsync()` — IDFV。アプリ再起動・OS アップデートで不変。同 vendor のアプリを全部消すと変わる
  - Android: `Application.getAndroidId()` — SSAID。再起動・再インストールでも不変（署名キー同一なら）。ファクトリーリセットでのみ変わる
- 権限不要・ATT プロンプト不要・App Store / Google Play のポリシー適合（非広告用途の識別子は許可）
- null が返る稀なケースに備え `expo-crypto` で UUID を生成し `expo-secure-store` にキャッシュする fallback を用意
- **生 ID はサーバに送らず、`sha256(device_id + server_secret)` をクライアント側（もしくはサーバ受信直後）でハッシュ化**して保存。漏洩時に元 ID が復元されず、プライバシーポリシー上も説明しやすい

### サーバ実装イメージ

新規テーブル:
```sql
CREATE TABLE ai_quota (
  key TEXT NOT NULL,        -- 'user:<id>' または 'device:<hash>'
  date TEXT NOT NULL,       -- 'YYYY-MM-DD' (JST)
  count INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY (key, date)
);
```

ミドルウェア（例）:
```
/api/dishes/:id/suggest などの AI 系ルートの直前に `rateLimitAi` を挟む
  - req.userId があれば key = 'user:<id>', limit = USER_AI_DAILY_LIMIT
  - なければ X-Device-Id ヘッダを必須化し key = 'device:<hash>', limit = GUEST_AI_DAILY_LIMIT
  - 日付（JST）で count++、超過したら 429 { error: 'ai_quota_exceeded', remaining: 0, resetAt: '...' }
  - 成功レスポンスにも X-AI-Remaining ヘッダで残数を返し、UI で残回数表示可能にする
```

AI エンドポイント自体は `requireAuth` から外し、`optionalAuth`（JWT があれば検証するが無くても通す）+ `rateLimitAi` の組み合わせに変える。

### クライアント UX

- 残回数を料理詳細画面などに表示（例: 「AI 提案 残り 2 回」）
- 上限到達時:
  - 未ログイン: 「今日の無料枠を使い切りました。ログインすると 1 日 20 回まで使えます [ログイン]」
  - ログイン済み: 「今日の上限に達しました。明日また使えます」
- エラー 429 を受け取った時もクライアント側で同じモーダル／トーストを出す

### 不正対策の補強（必要に応じて段階的に）

- Phase 1: 端末 ID ベースの回数制限のみ（本プラン）
- Phase 2: 1 回の呼び出しごとに `ai_requests` ログを残し、異常パターン（短時間連打・同一 IP からの多数デバイス）を検知
- Phase 3 (必要時): iOS App Attest / Android Play Integrity で「正規アプリからの呼び出し」を保証。本プランでは範囲外

## 案A〜E: ログイン不要 / 必須 の切り分け

※ いずれの案でも AI は「レート制限付きで未ログインでも呼べる」前提。以下の表の「ログイン必須」は**機能そのものにログインを要求するかどうか**を示す。

### 案A: 全機能ローカル動作（最もログイン圧が弱い）

| | ログイン不要 | ログイン必須 |
|--|-------------|-------------|
| 買い物リスト | ✅ 端末内に保存 | — |
| 料理 | ✅ 端末内に保存 | — |
| AI 具材・レシピ提案 | ✅（未ログイン制限：3回/日） | — |
| 自分のレシピ（いいね） | ✅ 端末内に保存 | — |
| みんなのレシピ閲覧 | ✅ | — |
| みんなのレシピへの公開 / いいね（集計反映） | — | ✅ |
| 複数端末間の同期 | — | ✅ |

- **Pros**: 体験は最高。アプリの価値をフルに体験してから登録できる
- **Cons**:
  - 買い物リスト / 料理 / 自分のレシピをオフラインローカルで持つには Zustand の永続化 + サーバスキーマをオプショナル userId にする大改修
  - TODO の「オフラインの時にローカルで変更を保存」と設計が重なるので実質的に同じ大きさの工事

### 案B: 閲覧と AI だけログイン不要（最小改修）

| | ログイン不要 | ログイン必須 |
|--|-------------|-------------|
| みんなのレシピ閲覧 | ✅ | — |
| AI 具材・レシピ提案 | ✅（未ログイン制限：3回/日、ローカルの料理名だけ渡す） | — |
| 買い物リスト / 料理 | — | ✅ |
| 自分のレシピ / いいね | — | ✅ |

- **Pros**:
  - サーバは `/api/saved-recipes/shared` と AI 系ルートを `requireAuth` から外し（代わりに rate limit）、それ以外は据え置き
  - モバイルは「みんなのレシピ」と「AI 試食」だけ未ログインで触れる
- **Cons**: アプリの中心機能である買い物リストを触れない。AI 結果を保存できないので使い捨て感が強い

### 案C: 書き込みはローカル、AI はレート制限付き共通、共有系はログイン必須（中道・推奨）

| | ログイン不要 | ログイン必須 |
|--|-------------|-------------|
| 買い物リスト | ✅ 端末内のみ | — |
| 料理（グルーピング・並べ替え） | ✅ 端末内のみ | — |
| みんなのレシピ閲覧 | ✅ | — |
| AI 具材・レシピ提案 | ✅ 3回/日（未ログイン） / 20回/日（ログイン済み） | — |
| 自分のレシピ保存 / いいね | — | ✅ |
| 端末間同期 | — | ✅ |

- **Pros**:
  - アプリの骨格（買い物リスト）と AI 体験がログインなしで触れる
  - ログインの動機が明確: 「AI 回数を増やしたい」「レシピを残したい」「別端末でも見たい」
  - 有料 AI はログイン有無に関わらず上限で守られるのでコスト暴走しない
- **Cons**:
  - 買い物リストを丸ごとローカル実装する必要あり。Zustand + AsyncStorage 永続化、サーバ API とのスキーマ互換を保つ
  - ログイン後に「ローカルのデータをアカウントに引き継ぎますか？」のマイグレーション導線が要る

### 案D: 匿名アカウント自動発行（サーバ改修が小さい）

| | 動作 |
|--|------|
| 初回起動時 | 端末 ID → 匿名ユーザーをサーバ側で作成、JWT を発行 |
| 全機能 | 通常通り使える（内部的には匿名ユーザー） |
| AI レート制限 | 匿名ユーザー種別で区別（例: `is_anonymous` フラグで制限を切り替え） |
| メール登録 | 任意タイミング。匿名ユーザーがメールと紐付き、既存データそのまま |

- **Pros**:
  - 既存コードへの侵襲が最小（モバイル / サーバともに認証ルートが変わらない）
  - データマイグレーションが不要（匿名 user_id をメール user に rename するだけ）
  - AI レート制限も既存の user_id ベースで一貫実装できる（端末 ID ベースの別系統を作らなくて済む）
- **Cons**:
  - 孤児アカウントが増える → 定期 GC が必要
  - App Store / Google Play のガイドラインで「登録なしで使える」要件は満たすが、内部的には全員ユーザー登録されている
  - 匿名 JWT が漏れた場合のデータ保護をどうするか
  - AI 制限の実効性は同じ（匿名ユーザーも端末 ID にバインドするので結局端末単位）

### 案E: デモモード（最も軽い）

| | ログイン不要 | ログイン必須 |
|--|-------------|-------------|
| ダミーの買い物リスト / 料理（操作はメモリ内のみ、次回起動でリセット） | ✅ | — |
| みんなのレシピ閲覧 | ✅ | — |
| AI 提案 | （表示のみ。実呼び出しはログイン必須 or デモは事前生成結果を出す） | ✅ |
| 保存 / いいね / 自分のレシピ | — | ✅ |

- **Pros**: 実装が極めて軽量。「雰囲気だけ見せて登録してもらう」
- **Cons**: 「触ったのに消える」体験はむしろ離脱を招く可能性。ストア審査向けの見せ札としてのみ価値

---

## 推奨案

**案C を推奨**（案Dを次善策）。

理由:
- AI をレート制限付きで未ログインにも開放することで、アプリの価値（買い物リスト + AI レシピ提案）を登録前に完全体験できる
- レート制限は user_id / device_id_hash の両対応で初期コストが平準化され、ログイン有無どちらでも暴走しない
- 買い物リストのローカル化は TODO 「オフライン対応」と同じ基盤で進められるので無駄にならない
- 案D（匿名アカウント）は短期実装は楽だが、孤児 GC と、匿名 JWT 管理の長期コストを考えると、クリーンに端末 ID ベースで切る案C の方が素直

## ログイン画面を表示するフロー（案）

現状: 起動 → `checkAuth` → 未認証なら即 `/(auth)/login` へ replace（`mobile/app/_layout.tsx:22-24`）。
ログイン不要モードを入れるなら、この「強制遷移」をやめて、**必要になった時だけモーダルで出す**のが自然。

### フロー案1: 必要になったらモーダル（推奨）

- 未ログインでも `(tabs)` に入れる
- ログイン誘導を出すトリガー:
  - **AI 未ログイン上限（3回/日）に到達** したとき → 「ログインすると 20回/日 まで使えます」
  - レシピ保存（いいね） / みんなに公開 を押したとき
- インラインのログインモーダル（メール → コード入力 2ステップ）を出す
- モーダルで成功したら元の操作を続行（例: AI 提案モーダルをそのまま開く）
- ヘッダーのメニュー（☰）に常時「ログイン」or「<email>」「ログアウト」を出す

**Pros**: 文脈が切れない、ログインの価値が直前のアクションと結びつく
**Cons**: 各機能の入口にゲート判定を仕込む必要がある

### フロー案2: 上部に常時バナー

- 未ログイン時、全画面の上部に「ログインすると AI が 20回/日 使えます [ログイン]」の細いバナー
- タップで `(auth)/login` に遷移
- ログイン必須操作を触ると案1と同じくモーダル or 画面遷移

**Pros**: ログインの存在が常に見える
**Cons**: バナーが常時見えてやや邪魔

### フロー案3: 初回オンボーディングで選ばせる

- 初回起動時に「ログインして始める / ログインせずに始める」の選択画面
- 以後は案1のモーダル or メニューからログイン

**Pros**: 意図がユーザー側で明確
**Cons**: 画面を 1 枚増やす。選択肢を見せる必要があるかは検討余地あり

### 推奨: 案1（必要になったらモーダル）+ メニューに常時ログインエントリ

- オンボーディング不要。未ログインでもすぐリストを触れる
- ゲート対象は「AI 上限到達時 / レシピ保存 / いいね / 公開」に絞れる

## 案C を採用する場合の実装論点

この節は案C が採択された場合の概算。個別タスクに分解するのは別途。

### サーバ側

- `/api/saved-recipes/shared` を `requireAuth` から外す（read-only の公開取得）
  - `getSharedRecipes` が `userId` を受け取って「その人のいいね状態」を付けている（`saved-recipes.ts:21-27`）。未ログイン時は liked フラグなしのバリアントが必要
- `/api/dishes/:id/suggest` などの **AI 系エンドポイントを `optionalAuth` + `rateLimitAi` に差し替える**
  - `optionalAuth`: Authorization ヘッダがあれば検証・`req.userId` をセット、無ければスルー
  - `rateLimitAi`: userId があればユーザー単位、無ければ `X-Device-Id` ヘッダ必須で端末ハッシュ単位にカウント
  - 上限値は環境変数で: `AI_LIMIT_USER=20`, `AI_LIMIT_GUEST=3`
- 新規テーブル `ai_quota(key, date, count)` を追加（上記「AI 呼び出しのレート制限設計」参照）
- 残回数レスポンスヘッダ `X-AI-Remaining` を付与
- `/api/shopping` `/api/dishes`（CRUD 系）`/api/saved-recipes` は `requireAuth` のまま

### モバイル側

- **最小改修版（推奨）**: 買い物リスト / 料理 / 自分のレシピのローカル保存は一旦スコープ外にして、まずは下記を実装する
  - 未ログインでタブに入れるようにする
  - みんなのレシピを未ログインでも閲覧可能にする
  - **AI 提案は未ログインでも呼べる**（3回/日）
  - AI 上限到達 or レシピ保存 を試みた時にログイン誘導モーダル
  - 端末 ID 取得と `X-Device-Id` ヘッダ付与
  - これは実質「案B + ログインゲートをモーダル化」
  - 買い物リストのローカル化は後続タスク（TODO の「オフラインの時にローカルで変更を保存」と統合）
- **フル版**: Zustand ストアに `mode: 'local' | 'server'` を持たせ、未ログインは AsyncStorage に保存、ログイン後に差分を PUSH してサーバと統合
  - マイグレーション UX: 「ローカルの 12 件をアカウントに移しますか？」

### 端末 ID 取得の実装詳細

- `expo-application` を追加（`npx expo install expo-application`）
- `src/utils/device-id.ts` を新規作成:
  - iOS: `Application.getIosIdForVendorAsync()`
  - Android: `Application.getAndroidId()`
  - null の場合は `expo-crypto` で UUID を生成し `expo-secure-store` にキャッシュ
  - 取得した生 ID を `sha256(raw_id + server_secret)` でハッシュ化して返す（secret は `app.config.js` の extra 経由）
- `src/api/client.ts` の axios インターセプタで、未ログイン時のみ `X-Device-Id` ヘッダを自動付与

### `_layout.tsx` の変更

- `RootNavigator` の未認証リダイレクトを廃止（`mobile/app/_layout.tsx:22-24`）
- `(auth)` グループは残す（明示的にログインしたい時 / モーダルから遷移する時に使う）
- `useAuthStore` に「ログインを要求する」ヘルパー（`requireLogin(onSuccess)` → モーダル表示 → 成功時に callback 実行）を追加

### Web 側

- 同様に起動直後のログイン強制を外す
- 保存系の UI は未ログイン時は「ログイン」ボタンに差し替え
- 端末 ID は **ブラウザでは** `localStorage` に UUID を発行して代用（IDFV/SSAID は使えない）
- Web は iOS/Android と仕組みが違うことを明記。PWA では localStorage がサイトデータ削除で消えるため、Web での制限は「ブラウザ単位・緩め」の想定

## 段階的ロールアウト提案

1. **Phase 1 (最小)**: 未ログインでタブを開ける + みんなのレシピ閲覧 + AI 提案（未ログイン 3回/日、ログイン済み 20回/日）+ ログイン必須操作はモーダル誘導 + メニューにログインエントリ
   - ストア審査向けの「登録なしでも中が見える / AI を試せる」を早く満たす
2. **Phase 2**: 買い物リスト / 料理のローカル保存。ログイン時はサーバに引き継ぎ
3. **Phase 3**: 自分のレシピのローカル保存（いいねはログイン必須のまま残す選択肢あり）
4. **Phase 4（必要時）**: App Attest / Play Integrity による正規アプリ検証で、レート制限の不正突破を抑止
